AI需求工作台 MVP 数据关系图

围绕“网页对话 → AI识别 → 候选需求 → 正式需求 → PRD → HTML原型 → 查询复用”的数据产生、影响和追溯关系。

会话与消息 需求资产 PRD与原型 AI配置与Trace 异常与反馈

1. MVP主数据流

数据从用户自然语言输入开始，经过后端编排和AI结构化处理，逐步沉淀为可查询、可追溯、可复用的需求资产。

用户输入网页对话框、/命令、按钮动作
→
conversation_message保存原始消息，先落库再调用AI
→
AI路由识别识别intent、target、nextActions
→
AI需求抽取输出结构化patch，不直接写库
→
requirement_candidate候选需求实时更新
→
requirement人工确认后生成正式需求
→
PRD / Prototype结构化PRD与HTML原型

3. 数据产生顺序

创建会话
用户点击“新建需求对话”，创建 conversation_session，状态为 empty。
保存用户消息
每次用户输入先写入 conversation_message，保证原始上下文不丢失。
AI路由识别
后端调用 intent_recognition Skill，生成 ai_trace，输出意图和建议动作。
AI需求抽取
后端根据路由结果调用 requirement_extract，输出候选需求创建或更新patch。
创建或更新候选需求
系统创建/更新 requirement_candidate，并将本次变化写入 requirement_candidate_patch。
完整度检查与追问
调用 completeness_check 和 reply_generate，更新候选需求缺失项并生成AI回复消息。
用户确认生成正式需求
候选需求达到阈值后，用户确认产品线、模块、标题，系统创建 requirement 和 requirement_version。
生成PRD
从正式需求调用 prd_generate，创建 prd 与多个 prd_section。
生成HTML原型
从已确认PRD调用 html_prototype_generate，创建 prototype 和 prototype_version。
进入查询与复用
需求、PRD、原型、会话和Trace写入 embedding_index，支持关键词和语义查询。

8. AI Trace贯穿关系

AI Trace不是业务结果本身，而是解释业务结果如何产生的证据链。所有AI调用必须写Trace。

Trace写入时机

  • 调用模型前生成trace_no。
  • 记录输入上下文、模型配置、Prompt版本。
  • 模型返回后记录输出、Token、耗时。
  • 失败时记录error_code和error_message。
  • 用户采纳或驳回时写入ai_action_feedback。

Trace反向追溯

  • 候选需求字段来自哪个AI抽取结果。
  • 完整度评分来自哪个Prompt版本。
  • PRD章节由哪个模型生成。
  • HTML原型由哪个Prompt和PRD章节生成。
  • AI输出是否被用户采纳。

10. MVP中的数据写入边界

AI可以做什么

  • 识别意图。
  • 抽取需求字段。
  • 判断是否拆分。
  • 生成候选需求patch。
  • 检查完整度。
  • 生成追问问题。
  • 生成PRD和HTML草稿。

后端必须做什么

  • 保存原始消息。
  • 校验AI输出JSON。
  • 决定是否执行nextActions。
  • 写入候选需求和patch。
  • 生成正式需求编号。
  • 控制状态流转。
  • 写入Trace和版本。

MVP不建议让大模型通过MCP直接写数据库。后续可以将固定业务接口封装为MCP工具，但仍不暴露通用SQL。
